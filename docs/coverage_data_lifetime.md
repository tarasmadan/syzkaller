# End-to-End Coverage Data Lifetime

This document describes how code coverage data is generated, collected, processed, and visualized in the `syzkaller` and `syzbot` ecosystem.

## 1. Generation (Kernel Space)

Coverage collection relies on compiler instrumentation and a dedicated kernel driver (KCOV).

### Compiler Instrumentation
When building the kernel with coverage support (typically via `CONFIG_KCOV=y`), the compiler (GCC or Clang) is instructed to insert a call to `__sanitizer_cov_trace_pc` at the beginning of every basic block (or CFG edge, depending on compiler and flags).

### KCOV (Kernel Coverage)
`KCOV` is the kernel component that records these calls.
- **Per-Task Buffer**: KCOV allocates a per-task buffer (array of unsigned longs) when requested by a user-space process via `ioctl`.
- **Trace Execution**: When a task executes an instrumented basic block, the `__sanitizer_cov_trace_pc` function is called. This function retrieves the current task's KCOV buffer and appends the Program Counter (PC) of the call site to it.
- **Shared Memory**: This buffer is shared between the kernel and user space via `mmap`, allowing efficient reading without the overhead of frequent system calls.

## 2. Collection (Executor)

The `syz-executor` is responsible for executing the test programs generated by `syz-fuzzer` and retrieving the coverage.

- **KCOV Setup**: `syz-executor` opens `/sys/kernel/debug/kcov` (or equivalent) and maps the shared buffer.
- **Enabling KCOV**: Before executing a test program, it enables KCOV mode (e.g., `KCOV_TRACE_PC`) for the current thread using `ioctl`.
- **Execution**: It runs the generated syscalls.
- **Retrieval**: After execution, it reads the number of recorded PCs from the first word of the shared buffer and then reads the PCs themselves.
- **Reporting**: The raw PCs are sent back to the `syz-fuzzer` (and subsequently to `syz-manager`) over the IPC link (pipes or sockets).

## 3. Processing and Merging (Manager)

The `syz-manager` orchestrates the fuzzing process and manages the global state, including coverage.

### Symbolization
The raw PCs received from the executor are just virtual addresses. To make them useful for humans and for analysis, `syz-manager` symbolizes them:
- **Database Creation**: On startup, `syz-manager` uses tools like `nm`, `objdump`, and `addr2line` on the kernel image (`vmlinux`) to build a database mapping PCs to functions, files, and line numbers.
- **Translation**: Raw PCs are looked up in this database to determine which source lines and functions were covered.

### Merging
- **Uniqueness**: Coverage is maintained as a set of unique PCs (typically a map in Go).
- **Global Corpus**: `syz-manager` maintains a global coverage map representing all code covered by the current corpus.
- **New Signal**: When a new program is executed, its coverage is compared against the global map. If it contains PCs not present in the global map, it is considered "interesting" and may be added to the corpus. The new PCs are merged into the global map.

## 4. Externalization and Visualization (Dashboard / Syzbot)

Coverage data is exposed and stored in several ways:

### Local Web UI
`syz-manager` serves a web interface that shows coverage reports. Users can browse directories and files to see covered and uncovered lines, colored appropriately.

### Raw Coverage Export
`syz-manager` exposes a `/rawcover` endpoint that dumps the raw covered PCs. This can be consumed by external tools like `syz-cover`.

### Syzbot / Dashboard
In the `syzbot` environment:
- **Reporting**: `syz-manager` instances report coverage statistics (and sometimes raw coverage data) to the central dashboard (`syzbot.appspot.com`) via `dashapi`.
- **Aggregation**: The dashboard aggregates coverage data from multiple managers running against different branches and configurations.
- **Display**: Syzbot displays coverage graphs over time and detailed coverage reports for specific builds, helping maintainers see the impact of fuzzing on different subsystems.
